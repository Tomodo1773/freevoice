# Tag スキル

Git タグ（セマンティックバージョニング）を付与し、GitHub にプッシュする。

## バージョン形式

`v{major}.{minor}.{patch}` （例: `v0.3.4`）

- **major**: アプリの機能が大幅に変わるとき
- **minor**: ちょっとした機能追加のとき
- **patch**: バグ修正などのとき

## 手順

1. 最新タグを取得し、**リモートも含めて**前回タグ以降の変更を洗い出す。直前の会話で触れた変更だけを根拠にバージョンを決めてはいけない（リモートには別経路でマージされた PR など、ローカルの会話に出てこない変更が含まれていることがある）:
   ```bash
   git fetch origin --tags
   git tag --sort=-v:refname | head -1
   git log <latest_tag>..origin/main --oneline
   ```
2. 上記コミットログから変更内容の全体像を把握し、ユーザーに「前回タグからこれだけの変更が入っています」と要約して提示する。そのうえで変更の種類を確認する（major / minor / patch）。ユーザーが既に指定している場合はその指定を尊重しつつ、コミットログと整合しないときは一度確認する。判断の目安:
   - 「バグ修正」「fix」「修正」のみ → patch
   - 「機能追加」「新機能」「feat」が含まれる → minor
   - 「大幅変更」「破壊的変更」「メジャー」 → major
3. バージョン番号を適切にインクリメントする。インクリメント時、上位が上がったら下位はゼロにリセットする（例: `v0.3.4` → minor なら `v0.4.0`）
4. 新しいタグをユーザーに確認する
5. 現在のブランチがmain以外の場合、変更をmainに反映する:
   - PRが未作成なら作成する
   - CIが実行中なら `gh pr checks <PR番号> --watch` で完了を待つ
   - CIが通ったら `gh pr merge <PR番号> --merge` でマージする
6. mainを最新にしてタグを付与する:
   ```bash
   git fetch origin main
   git tag v{new_version} origin/main
   ```
7. タグを GitHub にプッシュする:
   ```bash
   git push origin v{new_version}
   ```
8. 完了を報告する
